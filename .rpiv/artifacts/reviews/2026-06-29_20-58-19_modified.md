## 🟡 Important

### I1 🟡 Launch setup failure can strand a crawl run in `running`

**Where**
`apps/crawler/src/crawler.ts:230`

**Code**
```ts
  const { crawlRun } = startCrawlRun(config.db, {
```

**Why**
`startCrawlRun` inserts a non-terminal run before `launchContext: await createCloakLaunchContext(config.headless)` at `apps/crawler/src/crawler.ts:274`; that await is before the `try` block at `apps/crawler/src/crawler.ts:302` and before `finishCrawlRun` at `apps/crawler/src/crawler.ts:317`, so launch failures bypass finalization.

**Fix**
Move launch-context construction inside the guarded lifecycle or wrap the whole post-`startCrawlRun` path in `try/finally` that always attempts `finishCrawlRun`.

---

### I2 🟡 Finalization errors can mask crawler failures and skip cleanup

**Where**
`apps/crawler/src/crawler.ts:317`

**Code**
```ts
  finishCrawlRun(config.db, {
```

**Why**
`finishCrawlRun` executes before `await crawler.teardown().catch(() => undefined);` at `apps/crawler/src/crawler.ts:328` and before `throw runError;` at `apps/crawler/src/crawler.ts:331`. If finalization throws (for example, `apps/crawler/src/storage.ts:148` throws when a run is not found), the original crawler failure is masked and teardown is skipped.

**Fix**
Put teardown in a `finally`, preserve the original `runError`, and report finalization failure without replacing the root crawler failure.

### I3 🟡 Detail persistence can commit DB state without Dataset/counters reflecting it

**Where**
`apps/crawler/src/crawler.ts:124`

**Code**
```ts
  const stored = storeExtractedComic(config.db, {
```

**Why**
The SQLite transaction links the source entry to the crawl run before `await dataset.pushData(comic);` at `apps/crawler/src/crawler.ts:131`, and counters increment only afterward at `apps/crawler/src/crawler.ts:132`. A Dataset write failure can therefore leave persisted comic rows whose crawl-run `comicsStored`/`chaptersStored` counters do not include them.

**Fix**
Either push to Dataset before committing the DB row, or make the Dataset write diagnostic-only and increment DB-backed counters immediately after the DB transaction succeeds.

## 🔵 Suggestions

### Q1 🔵 Caller-owned SQLite handles can bypass foreign-key enforcement

**Where**
`packages/db/src/client.ts:25`

**Code**
```ts
  const sqlite = options.sqlite ?? createSqliteDatabase(options.fileName);
```

**Why**
`PRAGMA foreign_keys = ON` is only run in `createSqliteDatabase` at `packages/db/src/client.ts:19`. A caller-provided `sqlite` handle passes through `createDb` without the PRAGMA even though schema references such as `packages/db/src/schema.ts:87` rely on FK enforcement.

**Fix**
Run `sqlite.exec("PRAGMA foreign_keys = ON")` inside `createDb` for both owned and caller-provided handles.

### Q3 🔵 `tagsStored` is returned but never surfaced in crawl summaries

**Where**
`apps/crawler/src/types.ts:69`

**Code**
```ts
  tagsStored: number;
```

**Why**
`ComicStoredResult` includes `tagsStored`, but `addStoredResult` only increments chapters at `apps/crawler/src/crawler.ts:79`, and the final summary returns `chaptersStored` without a tag count.

**Fix**
Either add `tagsStored` to `ComicCrawlSummary` and counters, or remove it from `ComicStoredResult` if tags are intentionally not reported.

### Q4 🔵 New production lookup fields are missing supporting indexes

**Where**
`packages/db/src/schema.ts:87`

**Code**
```ts
    lastCrawlRunId: integer("last_crawl_run_id").references(() => crawlRuns.id, {
```

**Why**
The schema adds `crawl_runs.mode` and `comic_source_entries.last_crawl_run_id`, but current indexes cover `crawl_runs.source_id/status` and source-entry source/status/view-count fields, not those lookup columns. The production runner currently queries by `last_crawl_run_id` and by source/mode/latest-run.

**Fix**
Add indexes for the production query shapes, especially `comic_source_entries.last_crawl_run_id` and a source/mode/latest-run index on `crawl_runs` if full runs will be frequent.
